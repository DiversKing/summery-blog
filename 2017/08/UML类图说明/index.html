流云唯变

2017-08-25

UML类图说明

一. 概念

1.1. UML中描述对象和类之间相互关系的方式包括：

1). 依赖（Dependency）: 虚线箭头表示

  • 当类与类之间有使用关系时就属于依赖关系，不同于关联关系，依赖不具有“拥有关系”，而是一种“相识关系”，只在某个特定地方（比如某个方法体内）才有关系

2). 关联（Association）: 实线箭头表示

  • 表示类与类或类与接口之间的依赖关系，表现为“拥有关系”；具体到代码可以用实例变量来表示；

3). 聚合（Aggregation）: 带空心菱形头表示

  • 属于是关联的特殊情况，体现部分-整体关系，是一种弱拥有关系；整体和部分可以有不一样的生命周期；是一种弱关联；

4). 组合（Composition）: 带实心菱形头的实线表示

  • 属于是关联的特殊情况，也体现了体现部分-整体关系，是一种强“拥有关系”；整体与部分有相同的生命周期，是一种强关联；

5). 泛化（Generalization）: 空心箭头的实线

表示类与类之间的继承关系、接口与接口之间的继承关系

6). 实现（Realization）等： 空心箭头的虚线

  • 表示类对接口的实现

1.2. 其他概念

1). 类（Class）

  • 一般包含3个组成部分。第一个是类名；第二个是属性（attributes）；第三个是该类提供的方法（类的性质可以放在第四部分；如果类中含有内部类，则会出现第五个组成部分）。类名部分是不能省略的，其他组成部分可以省略。

2). 包（Package）

  • 包是一种常规用途的组合机制。UML中的一个包直接对应于Java中的一个包。在Java中，一个包可能含有其他包、类或者同时含有这两者。进行建模时，通常使用逻辑性的包，用于对模型进行组织；使用物理性的包，用于转换成系统中的Java包。每个包的名称对这个包进行了惟一性的标识

3). 接口（Interface）

  • 接口是一系列操作的集合，它指定了一个类所提供的服务。它直接对应于Java中的一个接口类型。

二. 详解

1. 依赖（Dependency）: 虚线箭头表示

元素A的变化会影响元素B，但反之不成立，那么B和A的关系是依赖关系，B依赖A；类属关系和实现关系在语义上讲也是依赖关系，但由于其有更特殊的用途，所以被单独描述。uml中用带箭头的虚线表示Dependency关系，箭头指向被依赖元素。

  • 要避免双向依赖。一般来说，不应该存在双向依赖;
  • 在 Java 或 C++语言中体现为局部变量、方法的参数或者对静态方法的调用。

  • 当类与类之间有使用关系时就属于依赖关系，不同于关联关系，依赖不具有“拥有关系”，而是一种“相识关系”，只在某个特定地方（比如某个方法体内）才有关系。

  • 方法参数示例： 人—-buy依赖—-> 车

2. 关联（Association）: 实线箭头表示

元素间的结构化关系，是一种弱关系，被关联的元素间通常可以被独立的考虑。uml中用实线表示Association关系，箭头指向被依赖元素。

  • 关联关系是类与类之间的联结，它使一个类知道另一个类的属性和方法。
  • 关联可以是双向的，也可以是单向的（#add还有自身关联）。双向的关联可以有两个箭头或者没有箭头，单向的关联有一个箭头。
  • 在 Java 或 c++中，关联关系是通过使用成员变量来实现的。
  • 表示类与类或类与接口之间的依赖关系，表现为“拥有关系”；具体到代码可以用实例变量来表示。（A类有一个成员变量保存的是B类的一个引用，也就是说由A类可以找到B类）

3. 聚合（Aggregation）：带空心菱形头表示

关联关系的一种特例，表示部分和整体（整体 has a 部分）的关系。uml中用带空心菱形头的实线表示Aggregation关系，菱形头指向整体。

  • 聚合是整体和部分之间的关系，例如汽车由引擎、轮胎以及其它零件组成。
  • 聚合关系也是通过成员变量来实现的。但是，关联关系所涉及的两个类处在同一个层次上，而聚合关系中，两个类处于不同的层次上，一个代表整体，一个代表部分。
  • 关联与聚合仅仅从Java或C++语法上是无法分辨的，必须考察所涉及的类之间的逻辑关系。
  • 属于是关联的特殊情况，体现部分-整体关系，是一种弱拥有关系；整体和部分可以有不一样的生命周期；是一种弱关联；

4. 组合（Composition）：带实心菱形头的实线表示

组合是聚合关系的变种，表示元素间更强的组合关系。如果是组合关系，如果整体被破坏则个体一定会被破坏，而聚合的个体则可能是被多个整体所共享的，不一定会随着某个整体的破坏而被破坏。uml中用带实心菱形头的实线表示Composition关系，菱形头指向整体。

  • 是关联关系的一种，是比聚合关系还要强的关系
  • 它要求普通的聚合关系中代表整体的对象负责代表部分的对象的生命周期。
  • 属于是关联的特殊情况，也体现了体现部分-整体关系，是一种强“拥有关系”；整体与部分有相同的生命周期，是一种强关联；

5. 泛化（Generalization）（继承） ：带空心箭头的实线表示

  • 通常所说的继承（特殊个体 is kind of 一般个体）关系，不必多解释了。uml中用带空心箭头的实线线表示Generalization关系，箭头指向一般个体。
  • 泛化（下图）表示一个更泛化的元素和一个更具体的元素之间的关系。泛化是用于对继承进行建模的UML元素。在Java中，用extends关键字来直接表示这种关系。
  • 泛化关系表示类与类之间的继承关系，接口与接口之间的继承关系。

6. 实现（Realization）：空心箭头的虚线表示

  • 元素A定义一个约定，元素B实现这个约定，则B和A的关系是Realize，B realize A。这个关系最常用于接口。uml中用空心箭头和虚线表示Realize关系，箭头指向定义约定的元素。
  • 实例（图I）关系指定两个实体之间的一个合同。换言之，一个实体定义一个合同，而另一个实体保证履行该合同。对Java应用程序进行建模时，实现关系可直接用implements关键字来表示。表达一种说明元素与实现元素之间的关系。

三. 比较

这几种关系都是语义级别的，所以从代码层面并不能完全区分各种关系；但总的来说，后几种关系所表现的强弱程度依次为：
　　
组合>聚合>关联>依赖

其中依赖（Dependency）的关系最弱，而关联（Association），聚合（Aggregation），组合（Composition）表示的关系依次增强。换言之关联，聚合，组合都是依赖关系的一种，聚合是表明对象之间的整体与部分关系的关联，而组合是表明整体与部分之间有相同生命周期关系的聚合。

而关联与依赖的关系用一句话概括下来就是，依赖描述了对象之间的调用关系，而关联描述了对象之间的结构关系。
